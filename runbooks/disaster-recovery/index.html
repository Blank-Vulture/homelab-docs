    障害復旧手順書¶

    前提条件¶

    • 3層バックアップアーキテクチャが稼働していること（ADR-003）
    • バックアップノード (VLAN 240) が正常に日次バックアップを取得していること
    • YubiKey / Nitrokey / オフラインUSB age鍵 のいずれかが手元にあること

    シナリオ別復旧手順¶

    シナリオ1: PVEノード1台の障害¶

    影響: サービス一部停止、Cephは冗長動作を継続

    復旧手順:
  1. 物理コンソールまたはvPro(DMZ)で原因調査
  2. ハードウェア障害 → 部品交換 or 予備機投入
  3. ソフトウェア障害 → 再起動で復旧を試行
  4. Ceph OSD回復: ceph osd tree で状態確認
  5. 該当ノードのDockerサービスを再起動

復旧時間: 30分〜2時間
データ損失: なし（Ceph冗長化で保護）

    シナリオ2: PVE全ノード同時障害（カーネルパニック）¶

    影響: 全サービス停止、Cephクラスタ停止

    復旧手順:

Phase 1: 状態確認 (10分)
  1. OPNsense WebUI (10.0.90.254) → ネットワーク正常性確認
  2. 各ノードに物理コンソール接続
     - PVE-DMZ-1: vPro (Intel AMT) でリモートKVM
     - PVE-3: 物理HDMI+KB接続
     - PVE-1/2: 物理HDMI+KB接続
  3. dmesg / journalctl でパニック原因を特定

Phase 2: 順次再起動 (30分〜1時間)
  優先順位:
    1st: PVE-3 (Ceph Monitor) → Cephクラスタの頭脳
    2nd: PVE-1, PVE-2 (Ceph OSD) → ストレージ実体
    3rd: PVE-GPU, PVE-DMZ-1 → クリティカルでないサービス

Phase 3: Ceph回復確認 (10分)
  $ ceph status
  $ ceph osd tree
  $ ceph health detail
  → HEALTH_OK or HEALTH_WARN (recovering) なら成功

Phase 4: サービス再起動 (30分)
  各ノードで docker compose up -d

復旧時間: 1〜3時間
データ損失: Ceph回復成功ならなし

    Ceph回復失敗の場合: → シナリオ4（バックアップからの完全復旧）へ

    シナリオ3: GitHub全情報漏洩¶

    影響: コード・設計情報・暗号化済みシークレットが攻撃者に公開

    即時対応 (30分以内):

  1. Cloudflare Dashboard
     - 既存トンネルを削除 → 新規トンネル作成
     - CF Access ポリシーで全サービスへのアクセスを一時ブロック

  2. GitHub
     - Settings → Developer Settings → Personal Access Tokens → 全revoke
     - Settings → SSH and GPG keys → 全削除 → 新しい鍵を登録
     - Repository Settings → Secrets → 全削除
     - 新しいリポジトリを作成し、バックアップノードからpush

  3. Tailscale
     - Admin Console → Devices → 不審なノードがないか確認
     - 必要に応じてauth keyをrevoke

  4. サービスのクレデンシャル回転
     ※ SOPSで暗号化済みのため平文は漏洩していないが、念のため回転:
     - Cloudflare Tunnel Token
     - Vaultwarden Admin Token
     - Grafana Admin Password
     - Pterodactyl DB Passwords

     sops secrets/lab.sops.yml  # 各トークンを再生成して保存
     # 各サービスの .env を更新して再起動

後続対応:
  5. 漏洩したネットワーク構成情報の評価
     - VLAN設計、IPアドレス体系 → 外部からは到達不可能なため即時リスクは低い
     - 必要に応じてIPアドレス体系を変更（優先度低）

  6. バックアップノードのGitHub認証情報を更新
     - 新しいdeploy keyを生成
     - rclone / git remote の設定を更新

データ損失: なし

    シナリオ4: PVE全滅 + Ceph壊滅（バックアップからの完全復旧）¶

    影響: 全サービス・全データ喪失、バックアップからの復旧が必要

    前提: バックアップノード (10.0.240.2) が正常稼働していること

Phase 1: バックアップノードからコード取得 (10分)
  管理PC から:
  $ ssh admin@10.0.240.2
  $ ls /srv/backup/git-mirrors/   # リポジトリ一覧確認
  $ ls /srv/backup/restic/        # バックアップ一覧確認

  管理PC で:
  $ git clone backup-node:/srv/backup/git-mirrors/homelab.git

Phase 2: Proxmox再インストール (2〜4時間)
  1. PVE-3: Proxmox VE インストール (USB起動)
  2. PVE-1: Proxmox VE インストール
  3. PVE-2: Proxmox VE インストール
  4. Cephクラスタ再構築 (空のクラスタ)
  5. ネットワーク設定 (VLAN trunk, bond)

Phase 3: データ復元 (30分〜1時間)
  バックアップノードから SSH で取得（管理PC経由）:

  # Vaultwarden DB
  $ ssh backup@backup-node
  $ restic -r /srv/backup/restic/vaultwarden snapshots  # 一覧確認
  $ restic -r /srv/backup/restic/vaultwarden restore latest \
      --target /tmp/restore/vaultwarden/

  # ゲームセーブデータ
  $ restic -r /srv/backup/restic/gamedata restore latest \
      --target /tmp/restore/gamedata/

  # Pterodactyl DB
  $ restic -r /srv/backup/restic/pterodactyl restore latest \
      --target /tmp/restore/pterodactyl/

  # 復元データを管理PCへ転送
  $ scp -r /tmp/restore/ admin-pc:/tmp/restore/

Phase 4: サービス再デプロイ (1〜2時間)
  1. SOPSシークレット復号 (YubiKeyタッチ)
     $ sops --decrypt secrets/lab.sops.yml > /tmp/.env.lab

  2. 各サービスの Docker Compose 起動
     $ docker compose -f docker/lab/vaultwarden/docker-compose.yml up -d
     $ docker compose -f docker/lab/forgejo/docker-compose.yml up -d
     $ docker compose -f docker/lab/monitoring/docker-compose.yml up -d
     $ docker compose -f docker/lab/pterodactyl/panel/docker-compose.yml up -d
     $ docker compose -f docker/lab/pterodactyl/wings/docker-compose.yml up -d
     $ docker compose -f docker/lab/cloudflared/docker-compose.yml up -d

  3. 復元データのマウント
     - Vaultwarden: sqlite3 DB を vaultwarden-data ボリュームにコピー
     - Pterodactyl: mysqldump をインポート
     - Wings: ゲームデータを /var/lib/pterodactyl/volumes/ にコピー

  4. Cloudflare Tunnel の再設定
     - 新しいトンネル作成 → トークン取得 → .env 更新

  5. Forgejo の復旧
     - バックアップノードの git mirror から初期リポジトリを復元

Phase 5: 動作確認 (30分)
  - Vaultwarden: ブラウザ拡張機能でログインテスト
  - ゲームサーバー: Minecraft接続テスト
  - 監視: Grafana ダッシュボード確認
  - 外部アクセス: Cloudflare Tunnel 経由で接続確認

合計復旧時間: 約4〜8時間
データ損失: 最大24時間分（前日のバックアップ時点まで復元）

    シナリオ5: GitHub全漏洩 + PVE全滅（同時発生）¶

    影響: 最悪のシナリオ。コード漏洩 + 全サービス・データ喪失

    対応順序:

  1. シナリオ3の即時対応を実行（クレデンシャル回転）
     → PVEが死んでいるのでサービス側の回転は後回し
     → Cloudflare/GitHub/Tailscale の外部サービスのみ先に対応

  2. シナリオ4の復旧手順を実行（バックアップからの完全復旧）
     → バックアップノードからコードとデータを取得

  3. 復旧後にクレデンシャル回転を完了
     → SOPSで新しいトークンを生成
     → 各サービスに適用

合計復旧時間: 約5〜9時間
データ損失: 最大24時間分

    シナリオ6: バックアップノード障害¶

    影響: バックアップ取得が停止（プライマリには影響なし）

    検知:
  - Discord/LINE通知が3日間来ない → dead man's switch 発動
  - VLAN 90 から SSH 接続不可

復旧:
  1. 物理確認（電源、SSD、ネットワークケーブル）
  2. ハードウェア故障 → 新しいPi/ミニPCをセットアップ
  3. Ansibleプレイブックで再構成
  4. Tier 3 (R2) から restic リポジトリを復元
     $ rclone sync r2:homelab-backup/restic/ /srv/backup/restic/
  5. git mirror を Forgejo から再取得
     $ git clone --mirror ssh://forgejo@10.0.200.11/repo.git

復旧時間: 1〜2時間（新HW調達済みの場合）
データ損失: なし（プライマリが稼働中）

    シナリオ7: 自宅物理災害（火災・水害）¶

    影響: 全物理デバイス喪失

    前提: Tier 3 (R2 + GitHub) にオフサイトバックアップが存在すること

復旧:
  1. 新しいハードウェアを調達
  2. R2からresticリポジトリをダウンロード
     $ rclone sync r2:homelab-backup/restic/ /tmp/restore/restic/
  3. GitHubからIaCコードをclone
  4. 以降はシナリオ4のPhase 2〜5に準じる

復旧時間: HW調達後 4〜8時間
データ損失: 最大24時間分

    バックアップ検証チェックリスト（月次）¶

    # 項目 手順 期待結果
    1 restic スナップショット一覧 restic snapshots daily 30 + weekly 12 + monthly 12 以内
    2 Vaultwarden DB 復元テスト restic restore latest → sqlite3 PRAGMA integrity_check ok
    3 ゲームデータ 復元テスト restic restore latest → tar 展開 → ファイル数確認 期待ファイル数と一致
    4 Git ミラー整合性 git fsck --full エラーなし
    5 R2 同期確認 rclone check /srv/backup/restic/ r2:homelab-backup/restic/ 差分なし
    6 ディスク残量 df -h /srv/backup/ 20% 以上の空き
    7 アラート通知テスト テスト通知送信 Discord/LINE に到達
