    ネットワークアーキテクチャ¶

    最終更新: 2026-03-07（ADR-008承認: CARP FW冗長化・VP6670追加・per-FW分配SW構成を正式採用）

    概要¶

    ゲーミング最適化・ゼロトラスト・将来拡張を目標としたホームラボネットワーク。 OPNsenseファイアウォールを中心に、MikroTikスイッチでVLANセグメンテーション、 Proxmoxクラスタでサービスを運用する。

    将来的な10Gbps WAN接続を視野に入れ、内部ネットワークは10Gbps SFP+基盤で構築。

    設計原則¶

    原則 説明
    ゼロトラスト VLAN間は明示的に許可した通信のみ通過（管理VLANも例外なし）
    ゲーム最優先 CAKE QoSでゲームトラフィックを最優先
    父の隔離 ASUS GT-AX11000で二重NAT + AIセキュリティ
    冗長化 Ceph 3ノード構成で1台故障しても継続。FW は CARP ペア（VP6670 MASTER + VP2440 BACKUP）で1秒以内フェイルオーバー
    将来拡張 10Gbps基盤で機器追加が容易、CPのCX-6 Lxは25GbE対応済み
    10Gbps WAN対応 両FWのSFP+を10GbE対応ポートで構成。10GbE WAN到来時はper-FW分配SW追加でSFP+2をWANに転用（ADR-009候補）
    3層バックアップ Lab (Ceph) + 独立バックアップノード + オフサイト (R2/GitHub)
    PULL型バックアップ バックアップノードは自ら取りに行く。PVEからバックアップノードへの通信は全拒否
    外部アクセス責務分離 CF ZT = データプレーン（サービス利用）、Tailscale = コントロールプレーン（インフラ管理）。責務を分離し一方の侵害が他方に波及しない設計。WANポート開放ゼロ
    管理操作分離 管理端末 (VLAN 90) を物理分離。ゲーミングPCに管理権限を一切置かない
    DMZ 物理分離 DMZ 専用スイッチを FW に直結。DMZ トラフィックがコアスイッチを通過しない物理分離構成
    OOB 完全分離 JetKVM + Yamahaスイッチ（別ベンダー）でOOB管理基盤を構築。MikroTik侵害がOOBに波及しない
    OOB エアギャップ VLAN 91/101 はインターネット非接続・VLAN 90 からの経路ゼロ。OOB管理端末は物理アクセス専用（ADR-007）
    BMC 一本化 ベンダーBMC（HP iLO等）を無効化しJetKVMに統一。BMC固有CVEリスクを排除

    多層防御¶

    1. Layer 1: Cloudflare（DDoS防御、WAF、ゼロトラストアクセス）
    2. Layer 2: OPNsense on Protectli VP6670+VP2440 CARP冗長ペア（ステートフルFW、IDS/IPS Suricata、CAKE QoS、coreboot、1秒以内フェイルオーバー）
    3. Layer 3: VLAN分離 + 物理分離（用途別ネットワーク、DMZ専用スイッチでFW直結、必要通信のみ許可）
    4. Layer 4: デバイスレベル（ASUS AIセキュリティ、Tailscale E2E暗号化、FIPS TPM）
    5. Layer 5: バックアップ隔離（VLAN 240、PULL型、append-only、整合性検証）
    6. Layer 6: OOB管理基盤（VLAN 91/101、別ベンダースイッチ、JetKVM、PXE自動復旧、完全エアギャップ）

    外部アクセスアーキテクチャ¶

    設計根拠: ADR-004

    データプレーン（サービス利用）とコントロールプレーン（インフラ管理）を異なるシステムで分離する。

    プレーン システム 用途 利用者
    データ Cloudflare Zero Trust Web UI、ゲーム接続、Grafana閲覧 自分・母・友人
    コントロール Tailscale (SaaS) SSH、PVE WebUI、Grafana管理 管理者のみ

    Tailscale 配置¶

    ノード Tailscale 用途
    全PVEベアメタル (5台) あり 1台でも生存すれば管理・復旧可能
    管理端末 (VLAN 90) あり クライアント
    Lab OOB管理端末 (VLAN 91) なし 物理アクセス専用（ADR-007）。VLAN 91 内でJetKVM/PXEに到達
    DMZ OOB管理端末 (VLAN 101) なし 物理アクセス専用（ADR-007）。VLAN 101 内でJetKVM/PXEに到達
    VM / LXC なし PVEコンソール経由で到達可能
    backup-node なし VLAN 240 隔離原則を維持

    分離の効果¶

    侵害シナリオ 影響範囲
    CF アカウント侵害 サービス経路のみ。管理プレーン（Tailscale）は無傷
    Tailscale アカウント侵害 管理経路のみ。ACL + SSH公開鍵認証で被害限定
    CF 大規模障害 サービス停止のみ。Tailscale で管理・復旧可能
    Tailscale 障害 既存トンネルは動作継続。新規接続のみ不可

    DMZ 物理分離アーキテクチャ¶

    設計根拠: ADR-005

    DMZ（VLAN 100）は外部公開サービスを収容する最高リスクゾーン。DMZ 専用スイッチを FW の SFP+ 2 に直結し、DMZ トラフィックをコアスイッチから完全に物理分離する。

    項目 値
    DMZ スイッチ CRS305-1G-4S+（SwOS モード）
    FW 接続 VP2440 SFP+ 2（専用インターフェース）
    運用モード SwOS（L2 スイッチングのみ、RouterOS CVE リスク排除）

    物理分離の効果¶

    侵害シナリオ 影響範囲
    コアスイッチ RouterOS 脆弱性 DMZ は無関係（コアスイッチを通過しない）
    DMZ ノードからの L2 横展開 FW を突破しない限り内部到達不可
    DMZ スイッチ侵害 DMZ セグメント内に限定

    3層バックアップアーキテクチャ¶

    設計根拠: ADR-003

    Tier 場所 内容 耐性
    1 Lab Ceph (VLAN 200) プライマリデータ ディスク1台故障
    2 バックアップノード (VLAN 240) PULL型バックアップ PVE全滅、ランサムウェア
    3 Cloudflare R2 + GitHub オフサイト複製 (Object Lock) 自宅物理災害

    通信方向の原則¶

    PVE (VLAN 200) ← SSH ── backup-node (VLAN 240) ── HTTPS → R2 / GitHub
                                                       (Tier 3)
PVE → backup-node: ██ 全拒否 ██ (OPNsense FWルール)
外部 → backup-node: ██ 全拒否 ██ (inbound ポート 0)
Mgmt → backup-node: SSH のみ (管理用)

    ランサムウェア耐性¶

    • バックアップノードのinboundポートはゼロ（管理SSH除く）
    • PVEノードはバックアップノードへの接続手段を一切持たない
    • バックアップデータはappend-only（backupユーザーは削除不可）
    • R2 Object Lockにより、オフサイトバックアップは書き込み後30日間削除不可
    • PULL後に毎回整合性検証（SQLite PRAGMA check、git fsck、サイズ異常検知）

    OOB管理詳細¶

    設計根拠: ADR-006, ADR-007

    Proxmoxクラスタ障害・侵害時でも管理操作を継続できる帯域外（OOB）経路を構築する。

    OOBネットワーク設計¶

    VLAN 名前 サブネット 用途
    91 Lab-OOB 10.0.91.0/24 Labゾーン帯域外管理（完全エアギャップ）
    92 OOB-Update 10.0.92.0/29 OOB管理端末のOS/FW更新専用WiFi（インターネット出口のみ）
    101 DMZ-OOB 10.0.101.0/24 DMZゾーン帯域外管理（完全エアギャップ）

    VLAN 91/101はインターネット非接続・完全エアギャップ。VLAN 90 からの有線ネットワーク経路もゼロ（ADR-007）。 OOB管理端末へのアクセスは物理アクセス専用。OS/FW更新時のみwlan0をVLAN 92（OOB-Update）に接続する。 VLAN 92はインターネット出口のみ許可し、VLAN 90を含む全内部VLANへの通信を全拒否する。

    OOB管理端末¶

    Lab・DMZゾーンそれぞれに物理管理端末を1台ずつ配置する（ADR-006）。

    ゾーン 役割 eth0（有線・常時） wlan0（更新WiFi・一時） 平常時アクセス
    Lab OOB管理端末 Lab OOBスイッチ経由で全LabノードのJetKVM/PXEに到達 VLAN 91 VLAN 92 物理アクセス専用（ADR-007）
    DMZ OOB管理端末 DMZ OOBスイッチ経由でDMZノードのJetKVM/PXEに到達 VLAN 101 VLAN 92 物理アクセス専用（ADR-007）

    管理端末間にKVMスイッチは設置しない。端末を共用するとゾーン間に経路が生まれるため、物理分離を維持する。

    OOBスイッチ: Yamaha SWX2310-10G¶

    コアネットワーク（MikroTik RouterOS）と異なるベンダーを採用することで、 MikroTik侵害後にOOBスイッチへ横展開できない構造を実現する。

    項目 Lab DMZ
    スイッチ Yamaha SWX2310-10G Yamaha SWX2310-10G
    運用モード L2スイッチング専用 L2スイッチング専用
    接続ノード PVE-1/2/3/GPU PVE-DMZ-1

    IP-KVM: JetKVM¶

    ノード 電源拡張品 VLAN
    PVE-1 ATX Extension Board 91
    PVE-2 ATX Extension Board 91
    PVE-3 (M75q Gen5) DC Power Control Extension 91
    PVE-GPU (HP Z8 G4) ATX Extension Board 91
    PVE-DMZ-1 (M90q Gen5) DC Power Control Extension 101
    • ベンダーBMC（HP iLO等）は無効化。JetKVMに一本化することで管理面を統一
    • JetKVM + PXEの組み合わせで現地作業ゼロのフルリモート再インストールを実現

    OOB障害耐性シナリオ¶

    侵害シナリオ 影響範囲
    MikroTik RouterOS 脆弱性悪用 OOBスイッチは別ベンダー（Yamaha）のため無関係
    Proxmoxクラスタ全滅 JetKVM + PXEでリモートから完全再インストール可能（物理アクセスで操作）
    HP iLO CVE JetKVMに一本化済みのためBMCは無効化済み
    VLAN 90（管理端末）侵害 VLAN 91/101 への経路ゼロ（ADR-007）。更新WiFi（VLAN 92）も内部VLAN全拒否のため無関係
    Tailscale アカウント侵害 OOB管理端末はTailscale非搭載。物理アクセス以外の侵入経路なし
