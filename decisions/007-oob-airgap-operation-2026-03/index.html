    ADR-007: OOB管理端末 完全エアギャップ運用とWiFi分離更新手順¶

    ステータス¶

    承認（2026-03-02）

    コンテキスト¶

    ADR-006でOOB管理基盤（JetKVM / PXEサーバー / Yamahaスイッチ）を決定した。 しかし、OOB管理端末へのアクセス方式が「VLAN 90 からのSSHのみ許可」と定義されており、 以下の構造的問題を抱えていることが判明した。

    問題1: OOB管理端末が「中間ホスト」になっている

    VLAN 90 管理端末 → SSH → OOB管理端末（VLAN 91）→ JetKVM / PXE

    OOB管理端末がエンドポイントではなくジャンプホストとして機能しており、 VLAN 90が侵害された場合、SSH経由でOOB管理端末に到達できてしまう。 OOB管理端末はフルLinux OSを持つため、そのOS全体が攻撃面になる。

    問題2: Yamahaスイッチの「ネットワーク到達経路ゼロ」優位性が活かされていない

    ADR-006でYamaha SWX2310-10Gを採用した主な理由のひとつは、 「MikroTik侵害後にOOBスイッチへ横展開できない構造」にある。 しかしVLAN 90 → VLAN 91 のSSH経路が存在する限り、 MikroTikがVLAN 90のトラフィックに介入できる場合はOOB管理端末に到達可能となる。

    問題3: 直接アクセス案の検討と棄却

    「OOB管理端末を廃止し、VLAN 90からJetKVMに直接アクセス（FWルールで制御）する」案も検討した。 しかしこの案では、VLAN 91 への新たなルーティングを開けることになり、 Yamahaスイッチが持つ「到達経路ゼロ」の優位性を構造的に失う。

    問題4: 更新時のインターネット接続の扱い

    完全エアギャップ運用では、JetKVMファームウェアおよびOOB管理端末OSの定期更新に インターネット接続が必要になる。 OPNsenseのFWルールを一時的に緩める案も検討したが、VLAN 91/101の隔離が ネットワークレベルで一時的に変化する点が設計原則に反する。

    代替として、OOB端末に搭載したWiFiインターフェースを用いた物理的に分離した更新経路を採用する。 更新中もVLAN 91/101のFWルールは一切変更せず、有線OOBネットワークの隔離は常時維持される。

    問題5: 更新WiFiをVLAN 90に乗せることによる管理端末との同一L2問題

    当初の設計ではwlan0をVLAN 90（Mgmt）に接続する案を採用していた。 しかし VLAN 90 は通常の管理端末（有線）も使用する同一L2セグメントであり、 更新ウィンドウ中にOOB端末と管理端末が同じブロードキャストドメインに共存する。

    IPフォワーディング無効化は「wlan0（VLAN 90）→ eth0（VLAN 91）への横断」を防ぐが、 「VLAN 90上の他デバイスがOOB端末のwlan0インターフェースに直接到達できる」ことは防がない。 更新ウィンドウ中に管理端末（VLAN 90）が侵害された場合、OOB端末に到達可能となる。

    専用VLAN（VLAN 92: OOB-Update）を新設し、インターネット方向のみ許可・VLAN 90内部通信を全拒否することで この問題を構造的に解消する。

    決定¶

    1. OOB管理端末を完全エアギャップ（物理アクセス専用）エンドポイントとして運用する¶

    VLAN 90 から VLAN 91 / VLAN 101 へのネットワーク経路をゼロにする。

    項目 変更前（ADR-006） 変更後（本ADR）
    VLAN 90 → VLAN 91 経路 SSH のみ許可 全拒否（常時）
    OOB管理端末の役割 中間ホスト（ジャンプホスト） エンドポイント
    アクセス方式 VLAN 90 からSSH 物理アクセス専用
    VLAN 91/101 FWルール 変動あり 静的・変更なし

    2. 更新手順: WiFiインターフェースによる物理分離更新¶

    OOB管理端末はWiFiアダプタを搭載する。通常は無効化し、更新時のみ有効化する。

    通常状態（エアギャップ）:

    OOB管理端末
  eth0: VLAN 91/101（有線、Yamaha OOBスイッチ接続）  ← 常時有効
  wlan0: 無効                                         ← 常時無効

    更新ウィンドウ:

    OOB管理端末
  eth0: VLAN 91/101（有線）   ← 変化なし
  wlan0: VLAN 92（OOB-Update）← 更新中のみ有効化

    Ubiquiti U7 ProにVLAN 92（OOB-Update）専用SSIDを設ける。 VLAN 92はインターネット方向のみ許可し、VLAN 90（管理端末）を含む内部VLANへの通信は全拒否する。 これにより更新中もOOB端末は管理端末のL2セグメントから完全に分離される。

    更新ウィンドウの手順:

    # 1. WiFiを有効化してVLAN 92 SSIDに接続
nmcli radio wifi on
nmcli connection up oob-update   # 事前に登録したVLAN 92接続プロファイル

# 2. 更新を実施
sudo apt update && sudo apt upgrade -y   # OOB管理端末OS
# JetKVM: wlan0経由でJetKVM Web UIにアクセスせず、eth0のVLAN 91経由のまま更新

# 3. 完了後、即座にWiFiを無効化
nmcli radio wifi off

    IPフォワーディングの無効化（必須設定）:

    eth0（VLAN 91）とwlan0（VLAN 90）間でOOB端末がルーターとして機能しないよう、 OSレベルでIPフォワーディングを無効化する。

    /etc/sysctl.d/99-oob-no-forward.conf
---
net.ipv4.ip_forward = 0
net.ipv6.conf.all.forwarding = 0

    3. Ubiquiti U7 Pro: VLAN 92 専用SSID¶

    項目 値
    SSID 非公開（hidden SSID）
    認証 WPA3-Personal
    VLAN 92（OOB-Update）
    用途 OOB管理端末の更新ウィンドウ専用
    クライアント分離 有効

    4. OPNsenseファイアウォールルールの変更¶

    ルール 変更内容
    VLAN 90 → VLAN 91 (SSH) 削除
    VLAN 90 → VLAN 101 (SSH) 削除
    VLAN 91 → 全VLAN / インターネット 引き続き全拒否（変更なし）
    VLAN 101 → 全VLAN / インターネット 引き続き全拒否（変更なし）
    VLAN 92 → インターネット (HTTP/HTTPS) 新規追加
    VLAN 92 → VLAN 90 / 他全内部VLAN 新規追加（全拒否）

    FWルールの静的維持がこの設計の最大の特徴である。 VLAN 92は「インターネット出口のみ持つ更新専用VLAN」として、VLAN 91/101と独立したルールセットで管理する。

    結果¶

    得られるもの¶

    • 常時エアギャップの実現: VLAN 91/101 への有線ネットワーク経路がゼロになる。 MikroTik侵害・VLAN 90 侵害のいずれが発生しても、OOB基盤に到達不可。
    • FWルールの静的維持: VLAN 91/101 のFWルールは更新時も含めて一切変更しない。 OPNsenseのルール変更ミスによる設定ミスリスクがゼロ。
    • Yamahaスイッチ優位性の完全活用: 有線OOBネットワークはインターネットと 物理的に接続されることがなく、「到達経路ゼロ」が構造として成立し続ける。
    • 更新の実用性確保: WiFiで直接インターネットに接続するため、 apt upgradeなど通常の更新手順をそのまま使える。
    • 攻撃面の最小化: OOB管理端末のSSHデーモンが不要になり、 有線側の常時リッスンポートがゼロ。

    トレードオフ¶

    • OOB管理端末はWiFiアダプタ（内蔵またはUSB）が必要
    • 更新中、OOB端末はeth0（VLAN 91/101）とwlan0（VLAN 92）を同時に持つ → IPフォワーディング無効化とiptablesでVLAN間ルーティングを構造的に阻止する
    • VLAN 92（OOB-Update）の新設: OPNsense / MikroTik / Ubiquiti それぞれに設定が必要
    • VLAN 92 SSID の認証情報をOOB端末に登録する必要がある
