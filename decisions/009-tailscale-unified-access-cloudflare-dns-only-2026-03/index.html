    ADR-009: 外部アクセス統合 — Tailscale一本化とCloudflare役割最小化¶

    ステータス¶

    承認

    コンテキスト¶

    ADR-004からの前提変化¶

    ADR-004（2026-02）は「友人・母がTailscaleを持っていない」という前提のもと、 Cloudflare Zero Trust（CF ZT）をデータプレーン、Tailscaleをコントロールプレーンとして 責務分離する設計を採用した。

    2026-03の運用を経て、以下の前提が変化した:

    前提（ADR-004策定時） 実態（2026-03）
    友人・家族がTailscaleを持っていない 全員Tailscaleクライアントを導入済み
    CF ZTがデータプレーンとして機能する Traefik LXCにcloudflaredを入れるとADR-004の責務分離に矛盾
    WARP + Tailscaleが共存できる MagicDNS干渉・Network Extension競合で事実上不可
    CF WAF/DDoSがサービスを保護する Tailscale側のサービスにはWAFが届かない構造

    発見された技術的問題¶

    1. cloudflaredの配置矛盾: CF TunnelをTailscale側LXCに入れると、 そのLXCがTailscaleとCF ZTの両方に依存する。CF侵害がTailscale側に波及するため ADR-004の責務分離原則に反する。

    2. WARP + Tailscale競合: Tailscale MagicDNS（100.100.100.100）と WARPのDNSプロキシが干渉し、CF_DNS_LOOKUP_FAILUREが恒常的に発生。 macOS Network Extension競合により同時利用は設計レベルで困難。

    3. ts.netの制約: tailscale cert が発行できるのは1台1ホスト名のみ。 サブドメイン（immich.machine.ts.net等）は作成不可。 複数サービスはポート番号管理になり運用性が下がる。

    4. CloudflareのTLS終端: CF Tunnel経由ではCloudflareがTLSを終端する。 Vaultwarden等のE2E暗号化サービスはアプリ層で保護されるが、 管理パネル操作はCloudflareから平文で見える。

    セキュリティ上の評価¶

    Cloudflare ZTのWAF・ボット対策は公開サービスに対して有効だが、 Tailscaleのみからアクセスするサービスには攻撃者が到達できないため意味をなさない。

    CF ZT:     インターネット → CF WAF → CF Tunnel → サービス
Tailscale: インターネット → (到達不可能) → サービス

    攻撃対象領域の観点では、WAFで防御するより到達不可能にする方が根本的に強い。

    決定¶

    1. 全サービスをTailscale subnet routing経由に移行¶

    CF TunnelおよびCF ZT（データプレーン）を廃止し、 全サービスへのアクセスをTailscaleのサブネットルーティング経由に統一する。

    【変更前】
インターネット → CF Tunnel → Traefik (10.10.0.101) → サービス

【変更後】
Tailscaleクライアント → PVEホスト（サブネットルーター） → Traefik (10.10.0.101) → サービス

    サービスアクセスフロー:

    Mac/iPhone（Tailscale ON）
  → WireGuard P2P or DERP
  → PVEベアメタル（Tailscale subnet router、10.10.0.0/24を広告）
  → Traefik LXC（10.10.0.101:443）
  → Let's Encrypt証明書でTLS終端
  → 各サービスLXC

    DNS設計:

    photos.rusticolus.dev  A  10.10.0.101  (Cloudflare Proxy: OFF)
vault.rusticolus.dev   A  10.10.0.101  (Cloudflare Proxy: OFF)
*.rusticolus.dev       →  必要に応じて追加

    内部IPを直接指定することで、Tailscale外からは名前解決できても到達不可能。

    2. TLS証明書をLet's Encrypt DNS-01に移行¶

    tailscale cert による ts.net 証明書から、 Let's Encrypt + Cloudflare DNS-01チャレンジによる *.rusticolus.dev 証明書に移行する。

    項目 変更前 変更後
    証明書発行 tailscale cert（ts.net） Let's Encrypt DNS-01（rusticolus.dev）
    証明書管理 手動更新 Traefikが自動更新
    サブドメイン対応 不可 ワイルドカード（*.rusticolus.dev）
    CF依存 なし DNS-01のみ（トラフィック通過なし）

    3. Tailscale配置方針（PVEベアメタルのみ）¶

    ADR-004セクション3の方針を継続・強化する。

    PVEベアメタル全台: Tailscale daemon + subnet router
  → 同一サブネットを複数PVEが広告 → 自動フェイルオーバー
  → PVE障害時でも他ノード経由でサービスに到達可能

LXC/VM: Tailscaleなし（subnet routing経由で到達）
  → 1LXC1サービスの原則を維持
  → Traefik LXCのTailscaleを削除

    将来PVEを追加するときのルール: 

    新PVEノード追加 → Tailscaleインストール → サブネット広告（当該VLANのサブネット）
以上。LXC/VMの設計は変わらない。

    4. CloudflareはDNSとDNS-01のみに限定¶

    機能 変更前 変更後
    DNSレコード管理 継続 継続
    Let's Encrypt DNS-01 一部利用 継続（Traefik統合）
    CF Tunnel 使用 廃止
    CF Zero Trust / WARP 使用 使用停止（設定は保持）
    CF Access ポリシー 使用 使用停止（設定は保持）

    CF ZTの設定（FIDO2ポリシー等）は削除せず保持する。 将来外部公開サービスが必要になった際に再利用可能。

    5. AdGuard Homeを追加（DNSフィルタリング）¶

    AdGuard HomeをLabクラスタにLXCとして追加し、 Tailscaleの組織DNS設定に登録する。

    Tailscale Admin Console → DNS → Custom Nameserver
  → 10.10.0.x（AdGuard Home LXC内部IP）
  → Override local DNS: ON

効果: 全Tailscaleデバイス（外出先のiPhoneを含む）のDNSがAdGuard Home経由になる

    6. CF ZT設定の保持と再利用ポリシー¶

    シナリオ 対応
    外部の人間とサービスを共有する必要が生じた CF Tunnelを選択的に追加、CF Access + FIDO2で保護
    MASQUE/QUICが必要なネットワーク環境（企業FW等） WARPを一時的に有効化（TailscaleのDERPで代替可能）
    一般インターネット通信の保護が必要 AdGuard Home（DNS）+ 必要ならWARPを追加

    結果¶

    セキュリティ上の改善¶

    観点 変更前 変更後
    攻撃対象領域 CF Tunnel経由でインターネットから到達可能 到達不可能（Tailscaleのみ）
    TLS終端 CloudflareがTLS終端（CF側で復号） Traefik（自前）がTLS終端
    CF侵害の影響 TunnelルーティングとAccessポリシーが改ざん可能 DNSレコード変更のみ（到達不可能は変わらず）
    設定複雑性 CF ZT + WARP + Access + Tunnelの多層設定 Tailscale ACL + Traefik設定のみ
    DNS over Tailscale なし AdGuard Homeで全デバイスのDNSフィルタリング

    多層防御の再定義¶

    Layer 1 (旧: Cloudflare WAF/DDoS)
  → Tailscale（WireGuard E2E + サブネットルーティング）
  → YubiKey FIDO2 + デバイス承認 + 90日key expiry + ACLロックダウン
  → 攻撃者はサービスに物理的に到達できない

Layer 2: OPNsense CARP + Suricata IDS/IPS（変更なし）

Layer 3: VLAN分離 + 物理分離（変更なし）

Layer 4: アプリ固有の暗号化
  → VaultwardenのBitwarden E2E暗号化
  → Traefik TLS（Let's Encrypt DNS-01）
  → CloudflareがTLSを終端しなくなる（改善）

Layer 5 (新規): AdGuard Home
  → 全Tailscaleデバイスのインターネット通信をDNSフィルタリング

Layer 6: バックアップ隔離 VLAN 240（変更なし）

Layer 7: OOB完全エアギャップ VLAN 91/101（変更なし）
  → Tailscale全滅時もOOBで復旧可能

    トレードオフ¶

    失うもの 影響 軽減策
    CF WAF/ボット対策 攻撃対象領域がゼロなので不要 —
    MASQUE（WARPのQUIC） DERPフォールバック（WebSocket/443）で代替 WARPを一時的に有効化
    CF ZTによるデバイスポスチャ高度機能 Tailscaleのデバイス承認・key expiryで代替 将来Tailscale Businessで強化可

    ADR-004との関係¶

    ADR-004の精神（責務分離・一方の侵害が他方に波及しない）は継続する。 ただし「CF ZT = データプレーン」という実装選択を、 実態（全ユーザーがTailscale持ち）に合わせて改訂する。

    ADR-004の責務分離の結果は本ADRでも維持される:

    侵害シナリオ ADR-004での影響 本ADR（009）での影響
    CF アカウント侵害 サービス経路のみ。管理プレーンは無傷 DNSレコード変更のみ。サービスは到達不可能のまま
    Tailscale アカウント侵害 管理経路のみ 全サービス経路が影響（ACL + MFAで軽減）
    CF 大規模障害 サービス停止のみ 影響なし（CFを通過しない）
    Tailscale 障害 既存接続は継続 既存接続は継続。新規接続のみ不可
    全Tailscale不能 Tailscale経由管理不可 OOB（VLAN 91/101）で復旧
