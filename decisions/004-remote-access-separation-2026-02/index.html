    ADR-004: 外部アクセス基盤の責務分離 — Cloudflare Zero Trust × Tailscale¶

    ステータス¶

    承認

    コンテキスト¶

    背景¶

    ホームラボの外部アクセス基盤として Cloudflare Zero Trust（以下 CF ZT）と Tailscale を併用しているが、両者の責務境界が曖昧であった。以下の戦略的選択肢を検討した:

    1. CF ZT 一元管理: Tailscale を廃止し CF ZT に統合
    2. Tailscale フォールバック化: CF 障害時のみ Tailscale を使用
    3. 責務分離: データプレーン（CF ZT）とコントロールプレーン（Tailscale）を明確に分離

    また、Tailscale の coordination server を自前で運用する Headscale への移行も検討した。

    現状の構成¶

    経路 用途 利用者
    CF Tunnel (HTTP) Web UI（VW, Forgejo, Panel） 自分・母・友人
    CF Tunnel (Private Network) + WARP ゲームサーバー接続 自分・母・友人
    Tailscale 管理アクセス（SSH, PVE WebUI, Grafana） 管理者のみ

    Tailscale は管理端末（VLAN 90）にのみインストールされており、サーバー側は PVE-3 のみ。

    脅威モデル¶

    外部アクセス基盤に対する攻撃ベクトルを分析した:

    カテゴリ 攻撃ベクトル CF ZT 一元管理時の影響 責務分離時の影響
    アカウント侵害 CF Dashboard 乗っ取り 全滅: サービス + 管理の全経路が掌握される サービス経路のみ。管理プレーンは無傷
    アカウント侵害 Tailscale アカウント乗っ取り N/A（廃止済み） 管理経路のみ。ACL + SSH公開鍵認証で被害限定
    サービス障害 Cloudflare 大規模障害 デッドロック: 障害を直すための管理経路も止まる サービス停止のみ。Tailscale で管理・復旧可能
    サービス障害 Tailscale coordination server 障害 N/A 既存トンネルは動き続ける。新規接続のみ不可
    中間者 CF Edge でのトラフィック傍受 管理 SSH のメタデータも CF に露出 SSH は Tailscale 経由（E2E, P2P）。CF に管理トラフィックは流れない
    中間者 Tailscale coordination server による不正鍵配布 N/A 理論上可能だがクライアント側で検知可。WireGuard の E2E 暗号化は維持
    供給網攻撃 cloudflared バイナリ改ざん サービス + 管理の全経路が侵害 サービス経路のみ
    供給網攻撃 Tailscale クライアント改ざん N/A 管理経路のみ。被害はPVEベアメタルへのSSHに限定

    最大のリスク: CF ZT 一元管理における CF アカウント侵害

    CF Dashboard を奪取した攻撃者は以下が可能: - Tunnel のルーティングを改ざんし、内部サービスを攻撃者のサーバーに向ける - Access Policy を変更し、攻撃者自身にアクセスを付与 - 管理 SSH 接続先を書き換え、MITM で認証情報を窃取

    責務が分離されていれば、CF が完全に侵害されても管理プレーン（Tailscale）は独立した認証・経路で動作し続ける。

    決定¶

    1. データプレーンとコントロールプレーンの分離¶

    2つのシステムは対等な「2つのVPN」ではなく、異なるレイヤーの責務を持つ:

    ┌──────────────────────────────────────────────────────┐
│  データプレーン: Cloudflare Zero Trust                 │
│                                                      │
│  ・Web UI (Vaultwarden, Forgejo, Pterodactyl Panel)  │
│  ・ゲームサーバー接続 (WARP + Private Network)         │
│  ・Grafana 閲覧 (read-only ダッシュボード)             │
│                                                      │
│  利用者: 自分・母・友人                                │
│  経路: クライアント → CF Edge → cloudflared → サービス  │
│  特性: CF Edge が中間者、HTTP/TCP プロキシ              │
└──────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────┐
│  コントロールプレーン: Tailscale                       │
│                                                      │
│  ・SSH (PVE ベアメタル緊急管理)                        │
│  ・PVE WebUI (8006)                                  │
│  ・Grafana 管理 (設定変更、アラートルール編集)           │
│  ・CF 障害時のフォールバック                            │
│                                                      │
│  利用者: 管理者のみ (管理端末 VLAN 90)                  │
│  経路: 管理端末 ←→ PVE (P2P WireGuard)                │
│  特性: E2E 暗号化、P2P 直接接続、中間者なし             │
└──────────────────────────────────────────────────────┘

    2. Tailscale SaaS を維持（Headscale は不採用）¶

    Headscale（自前 coordination server）を検討したが、不採用とする。

    Headscale の鶏と卵問題¶

    配置先 問題
    PVE クラスタ上の VM PVE 障害 → Headscale 死亡 → PVE に管理接続不能 → 復旧不能。コントロールプレーンの存在意義が消滅
    backup-node (VLAN 240) inbound ゼロの隔離原則に違反。バックアップノードでサービスを公開する設計矛盾
    管理端末 (RPi) RPi 単体故障で管理経路が全消失。SPOF そのもの
    外部 VPS コスト増 + VPS 自体の管理負荷 + VPS 侵害で tailnet の制御権を失う

    Tailscale をコントロールプレーンに使う目的は「自分のインフラが壊れた時でも管理経路を確保する」こと。その管理経路の依存先が自分のインフラ上にあったら、目的と矛盾する。

    WireGuard の設計が Tailscale SaaS のリスクを軽減¶

    懸念 実態
    Tailscale 社にメタデータが見える 見えるのは「デバイスX が 時刻Y に IPアドレスZ から接続」程度。通信内容は WireGuard E2E 暗号化で復号不可。ホームラボの接続メタデータは実害が極めて小さい
    Tailscale 社が悪意の鍵を push 理論上は coordination server が不正ピアの公開鍵を配布可能。ただしクライアント側で検知可能であり、Tailscale 社のビジネスを破壊する行為。現実的脅威ではない
    coordination server 障害 既存の WireGuard トンネルは動き続ける。coordination server は鍵交換の仲介のみで、確立済み接続には不要
    アカウント乗っ取り 本ADRのセクション6「アカウント強化」で対策

    Headscale が意味を持つケース（本環境には該当しない）¶

    • デバイス 100台超で Tailscale 無料枠を超過する場合
    • 規制環境で外部 coordination server を使用できない場合
    • coordination server を完全に自己管理したい知的好奇心（ただしコントロールプレーンの可用性と引き換え）

    3. Tailscale のサーバー側配置: 全 PVE ベアメタル¶

    配置案 利点 欠点
    PVE-3 のみ（現状） 攻撃面最小 PVE-3 障害で管理不能
    全 PVE ベアメタル（採用） どれか1台生存で管理可能 攻撃面が増加（ACLで軽減）
    専用ジャンプホスト VM 隔離・ハードニング可 VM は Ceph 依存。全 PVE 障害に脆弱

    全 PVE ベアメタルに Tailscale を配置する根拠:

    • 「PVE 全台カーネルパニック」は ADR-003 で想定済みの脅威。だが「1台だけ生存」の確率のほうが高く、その1台から復旧を開始できる
    • ベアメタルなので Ceph / VM に依存しない
    • Tailscale ACL で管理端末以外からの接続を拒否すれば、攻撃面の増加は最小限
    • VM / LXC には入れない（PVE コンソール経由で到達可能なため不要）

    配置対象:

    ノード Tailscale 理由
    PVE-1 (CPU ワーカー) あり ベアメタル緊急管理
    PVE-2 (CPU ワーカー) あり ベアメタル緊急管理
    PVE-3 (Lab CP) あり（既存） ベアメタル緊急管理
    PVE-GPU (HP Z8 G4) あり ベアメタル緊急管理
    PVE-DMZ-1 (M90q) あり DMZ ノード緊急管理
    管理端末 (VLAN 90) あり（既存） クライアント
    VM / LXC なし PVE コンソール経由で到達可能
    backup-node なし VLAN 240 の隔離原則を維持
    ゲーミング PC なし 管理権限を一切置かない（ADR-003）

    4. Tailscale ACL のロックダウン¶

    Tailscale ACL を最小権限に設定する:

    {
  "acls": [
    // 管理端末 → PVE ベアメタル: SSH + PVE WebUI のみ
    {
      "action": "accept",
      "src": ["tag:mgmt"],
      "dst": [
        "tag:pve:22",      // SSH
        "tag:pve:8006"     // Proxmox WebUI
      ]
    },
    // 管理端末 → PVE ベアメタル: Grafana (設定変更時のみ)
    {
      "action": "accept",
      "src": ["tag:mgmt"],
      "dst": ["tag:pve-cp:3000"]  // Grafana on PVE-3
    }
  ],

  // 暗黙の deny-all: 上記以外は全て拒否

  "tagOwners": {
    "tag:mgmt": ["autogroup:admin"],
    "tag:pve":  ["autogroup:admin"],
    "tag:pve-cp": ["autogroup:admin"]
  },

  "nodeAttrs": [
    {
      // 全ノードの鍵有効期限を 90 日に強制
      "target": ["*"],
      "attr":   ["autoApprovers:exitNode=false"]
    }
  ]
}

    ポイント: - PVE ノード間の Tailscale 経由通信は許可しない（VLAN 200/250 で直接通信する） - 管理端末以外のデバイスが tailnet に参加しても、ACL で到達先がゼロ - Exit Node 機能は無効化（Tailscale 経由でインターネットに出る必要がない）

    5. 日常運用と緊急時の使い分け¶

    場面 経路 理由
    Grafana でダッシュボード閲覧 CF Tunnel 日常的な read-only 操作
    Grafana のアラートルール変更 Tailscale インフラ設定変更は管理プレーン
    Vaultwarden のパスワード利用 CF Tunnel 日常的なサービス利用
    Forgejo でコード閲覧・push CF Tunnel 日常的なサービス利用
    PVE WebUI で VM 操作 Tailscale ハイパーバイザ管理
    SSH でサーバーメンテナンス Tailscale インフラ管理
    CF 障害時のサービス確認 Tailscale → PVE → VM フォールバック
    PVE 障害時の緊急復旧 Tailscale → 生存ベアメタル 最終手段
    ゲームサーバー接続（自宅） VLAN 10 直接 最低レイテンシ
    ゲームサーバー接続（外出先） CF WARP データプレーン
    友人のゲーム接続 CF WARP / playit.gg データプレーン

    6. Tailscale アカウントの強化¶

    Tailscale SaaS を維持する以上、アカウント侵害が最大のリスク。以下で対策:

    対策 説明
    MFA をハードウェアキーに限定 FIDO2/WebAuthn (YubiKey)。TOTP・SMS は無効化
    デバイス承認を手動に設定 auto-approve 無効。新デバイス追加には管理者の明示的承認が必要
    Key expiry を有効化 90日ごとに再認証を強制。盗まれた鍵の有効期間を限定
    Tailscale SSH を無効化 SSH 認証は OS レベルの公開鍵認証で行う。Tailscale の SSH 機能に依存しない
    Auth key をリポジトリに保存しない Tailscale の auth key は SOPS 暗号化で管理

    7. Grafana の CF Tunnel 公開（新規）¶

    Grafana を CF Tunnel 経由でも閲覧可能にする。日常的なダッシュボード確認を CF 経由に移行し、Tailscale は管理操作（設定変更）時のみ使用する。

    項目 値
    ホスト名 grafana.example.com
    内部サービス http://10.0.200.31:3000
    CF Access Policy Admins のみ
    Grafana 側設定 CF 経由アクセスは Viewer ロール（read-only）
    管理操作 Tailscale 経由の直接アクセスで Admin ロール

    結果¶

    • 責務分離: データプレーン（CF ZT）とコントロールプレーン（Tailscale）を明確に分離。一方の侵害が他方に波及しない
    • CF 障害耐性: CF 大規模障害時も Tailscale 経由で全 PVE ノードを管理可能。デッドロック回避
    • CF アカウント侵害耐性: CF Dashboard が乗っ取られても、管理プレーン（Tailscale）は独立した認証・経路で動作。攻撃者は管理 SSH に到達不可
    • PVE 部分障害耐性: 全ベアメタルに Tailscale を配置することで、1台でも生存すればそこから復旧開始可能
    • Headscale 不採用: 鶏と卵問題により、コントロールプレーンの依存先を自インフラに置く矛盾を回避
    • 運用負荷: Tailscale SaaS のため coordination server の運用不要。ACL 設定のみ管理
    • コスト: Tailscale 無料枠内（デバイス数 < 100、ユーザー数 < 3）
    • Grafana 二重公開: 日常閲覧は CF Tunnel（Viewer）、管理操作は Tailscale（Admin）で権限分離
