    ADR-011: Docker Compose サービス群を LXC ネイティブ実行に移行¶

    • 日付: 2026-03-26
    • ステータス: 承認済み
    • 関連: ADR-009（Tailscale統一アクセス）, ADR-010（AI自律管理基盤）

    背景¶

    現在、vaultwarden・Prometheus・Grafana・Loki・AdGuard Home・Jellyfin が docker-node VM (10.10.0.107) 上の Docker Compose で稼働している。この構成は以下の問題を持つ:

    1. VM→Docker の二重の仮想化オーバーヘッド
    2. アップデートワークフローが未整備（weekly-update.yml に統合されていない）
    3. 1サービス1LXC原則（既存の immich/traefik/tailscale/postgres で確立済み）に反する

    決定事項¶

    以下の方針でサービスを LXC に移行する:

    移行対象と推奨構成¶

    サービス LXC OS RAM Disk IP 管理 根拠
    AdGuard Home Alpine 3.23 256 MB 2 GB 10.10.0.105 OpenRC Go バイナリ、常時稼働DNS
    Vaultwarden Alpine 3.23 256 MB 2 GB 10.10.0.106 OpenRC APK パッケージあり、256MBで動作
    Prometheus Alpine 3.23 256 MB 4 GB 10.10.0.108 OpenRC APK パッケージあり
    Grafana Alpine 3.23 256 MB 4 GB 10.10.0.109 OpenRC APK パッケージあり
    Loki Alpine 3.23 256 MB 2 GB 10.10.0.110 OpenRC APK パッケージあり
    Jellyfin Ubuntu 24.04 2048 MB 20 GB 10.10.0.111 systemd .NET は musl 非対応、/dev/dri GPU パススルー必要
    Vault Debian 12 512 MB 10 GB 10.10.0.112 systemd HashiCorp APT リポジトリ（ADR-010 Phase 1）
    Authentik Debian 12 2048 MB 20 GB 10.10.0.113 Docker Compose-in-LXC 公式はDocker Compose推奨。LXC nesting=true で対応（ADR-010 Phase 2）

    例外: Docker を継続使用するもの¶

    • Pterodactyl Panel + Wings: Wings がゲームサーバーを Docker コンテナとして起動する設計。Docker 不可避。docker-node VM はゲームサーバー専用として維持。
    • Authentik: 公式サポートが Docker Compose のみ。LXC 内で Docker を使用（nesting=true）。これは "VM の中で Docker" ではなく "LXC の中で Docker" であり、オーバーヘッドは許容範囲。

    データ移行手順（docker-node 停止中のため）¶

    1. docker-node (10.10.0.107) を一時起動
    2. 各サービスのデータをエクスポート:
    3. Vaultwarden: docker cp vaultwarden:/data/ /tmp/vaultwarden-backup/
    4. AdGuard Home: docker cp adguardhome:/opt/adguardhome/conf/ /tmp/adguard-backup/
    5. Prometheus: データは短期メトリクスのため再構築可能（任意マイグレーション）
    6. Grafana: docker cp grafana:/var/lib/grafana/ /tmp/grafana-backup/
    7. Loki: データは再収集可能（任意マイグレーション）
    8. 各 LXC を起動後、バックアップを対応パスに復元
    9. Vaultwarden は最優先でマイグレーション（パスワードデータ消失は致命的）

    ADR-010 実装順序¶

    1. Phase 1: Vault LXC 構築 → Raft バックエンド初期化 → SOPS+age をフォールバックとして維持
    2. Phase 2: Authentik LXC 構築 → OIDC 設定 → 各サービスの SSO 統合
    3. Phase 3: Teleport LXC 構築 → SSH アクセスを Teleport 経由に段階移行

    代替案¶

    • Kubernetes (k3s) で全サービスを管理: k3s はすでに Forgejo 専用クラスタとして稼働中。ステートフルサービス（Vaultwarden、DB 等）をコンテナ化するとデータ永続化の複雑性が増す。1サービス1LXCの方がシンプル。
    • Docker Compose を VM ではなく LXC に移動: オーバーヘッドを削減できるが、根本的な問題（更新自動化・ライフサイクル管理）は解決しない。

    影響¶

    • docker-node VM: Pterodactyl+Wings 専用に再定義。メモリ/CPU を大幅削減可能
    • 週次アップデート: 各 LXC が独自の update-*.yml を持ち、weekly-update.yml から呼ばれる
    • Traefik ルール: 各新サービスの Traefik ルールファイル追加が必要
