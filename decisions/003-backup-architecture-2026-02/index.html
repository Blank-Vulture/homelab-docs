    ADR-003: ゼロトラスト・バックアップアーキテクチャ¶

    ステータス¶

    承認

    コンテキスト¶

    背景¶

    ホームラボのデータ保護に関して、以下の同時障害シナリオへの耐性が欠如していた:

    1. GitHub全情報漏洩: リポジトリ・CI/CD設定・暗号化済みシークレットが攻撃者に公開
    2. Proxmox全ノード同時障害: 共通カーネル脆弱性による全台カーネルパニック
    3. ランサムウェアによるLab内横展開: PVEノード侵害からCephストレージ暗号化

    現状の問題¶

    • Dockerボリューム・データベースのバックアップが一切存在しない
    • Ceph 3重複製は「1台故障」への耐性のみ。全台同時停止は想定外
    • Vaultwardenパスワードデータベースの消失は自分+母の全デジタルライフに影響
    • ゲームサーバーのワールドデータ（Minecraft等）は再現不可能
    • VLAN 90（管理VLAN）が全VLANへ無制限アクセス可能な設計上の欠陥

    脅威モデル¶

    バックアップシステムおよびインフラ全体に対する攻撃ベクトルを包括的に分析した:

    カテゴリ 攻撃ベクトル リスク
    ネットワーク VLANホッピング（二重タギング） バックアップノードへの直接攻撃
    ネットワーク ARP スプーフィング L2通信の傍受・改ざん
    ネットワーク VLAN 90 全許可ルール経由の到達 管理VLAN侵害で全VLAN崩壊
    ネットワーク Tailscaleが全VLAN分離をバイパス 端末感染で全管理アクセス奪取
    ホスト侵害 PUSH型バックアップの書き込み権限悪用 バックアップの暗号化・削除
    ホスト侵害 backup-export.sh改ざん（スロー・ポイズニング） 全retention window汚染
    ホスト侵害 バックアップノード自体の侵害 SSH鍵・R2クレデンシャル窃取
    端末感染 ゲーミングPC感染→Tailscale→全管理権限 全インフラ掌握（最大リスク）
    端末感染 母のスマホ感染→VW master PW窃取 母のパスワード全漏洩
    端末感染 母のスマホ/PC→ゲームサーバー脆弱性→Docker escape PVE root取得
    端末感染 会社PC→VLAN 20橋渡し→会社⇔自宅の双方向リスク 会社インシデントが波及
    サプライチェーン Docker image汚染 rootプロセスでの任意コード実行
    物理 バックアップノードの窃取 SSH鍵・バックアップデータ露出
    運用 バックアップの静かな失敗 災害時に復元データなし
    運用 NTP操作によるretention計算狂い 正常バックアップの早期削除

    決定¶

    1. PULL型バックアップアーキテクチャ¶

    PUSH型（PVEノードがバックアップサーバーに送信）ではなく、PULL型（バックアップノードがPVEノードからデータを取得）を採用する。

    根拠: PUSH型ではPVEノードがバックアップサーバーへの書き込み権限を持つため、PVE侵害時にバックアップも破壊される。PULL型ではPVEノードはバックアップノードへの接続手段を一切持たない。

    2. 専用VLAN 240 (Backup) の新設¶

    項目 値
    VLAN ID 240
    名前 Backup
    サブネット 10.0.240.0/29
    ゲートウェイ 10.0.240.1 (OPNsense)
    バックアップノード 10.0.240.2
    DHCP なし（静的のみ）
    物理スイッチ CRS328（アクセススイッチ）
    セキュリティレベル 最高（隔離、inboundゼロ）

    /29を選択した理由: バックアップノードは1台のみ。/24 (254ホスト)は過剰。/29 (6ホスト)で将来の冗長化にも対応しつつ、攻撃面を最小化。

    CRS328を選択した理由: PVEノードが接続されるCRS309とは物理的に異なるスイッチ。VLANホッピングがスイッチ境界で遮断される。

    3. ファイアウォールルール¶

    バックアップノードのinboundポートはゼロ（管理SSH除く）:

    • OUT: バックアップノード → PVEノード (10.0.200.1-3): SSH (22/tcp) のみ
    • OUT: バックアップノード → Forgejo (10.0.200.11): SSH (22/tcp) のみ
    • OUT: バックアップノード → Internet: HTTPS (443/tcp)、DNS (53)、NTP (123) のみ
    • IN: VLAN 90 → バックアップノード: SSH (22/tcp) のみ（管理用）
    • IN/OUT: その他全て拒否

    4. VLAN 90（管理VLAN）のルール修正¶

    既存の「全VLANへの全許可」を「明示的許可のみ」に変更。これにより管理VLANが侵害されてもバックアップVLANを含む各VLANへの横展開が制限される。

    5. SSH制限（command=強制 + ホスト鍵ピンニング）¶

    PVEノード側の authorized_keys: 

    command="/usr/local/bin/backup-export.sh",no-port-forwarding,no-X11-forwarding,no-agent-forwarding,no-pty ssh-ed25519 AAAA...

    • backup-export.sh と authorized_keys は chattr +i（immutable）で保護
    • SSH ホスト鍵は known_hosts にピンニング（MITM防止）

    6. Append-Only ストレージ + 長期保持¶

    • バックアップデータディレクトリは chattr +a（append-only）
    • backupユーザーは新規追加のみ可能、既存データの変更・削除は不可
    • restic forget（古いスナップショット削除）はrootのsystemd timerが週1回のみ実行
    • 保持ポリシー: daily 30日 + weekly 12週 + monthly 12ヶ月
    • スロー・ポイズニングで汚染できるのはdailyのみ。weekly/monthlyに正常データが残る

    7. 整合性検証（PULL後に毎回実行）¶

    データ種別 検証方法
    Vaultwarden SQLite PRAGMA integrity_check + レコード数の期待範囲チェック
    ゲームセーブ tar ファイル数・合計サイズの前回比較 (±50%超で警告)
    Pterodactyl SQL SQL構文チェック + テーブル数検証
    Git リポジトリ git fsck + git bundle verify

    8. 3層バックアップ構造¶

    Tier 場所 内容 耐性
    1 Lab Ceph (VLAN 200) プライマリデータ ディスク1台故障
    2 バックアップノード (VLAN 240) PULL型バックアップ PVE全滅 + ランサム
    3 Cloudflare R2 + GitHub オフサイト複製 自宅火災

    R2はObject Lock (WORM) を有効化し、バックアップノードのクレデンシャルが漏洩しても既存オブジェクトを削除不可にする。

    9. バックアップノードのハードニング¶

    • OS: Debian 12 Stable（PVEカーネルとは異なる系統）
    • CPU: ARM64推奨（x86 PVEとのアーキテクチャ多様性確保）
    • ディスク: LUKS2フルディスク暗号化（TPM-backed自動起動）
    • Dockerなし: restic, git, rclone, sqlite3はネイティブインストール
    • USBGuard: 許可デバイス以外をブロック
    • unattended-upgrades: セキュリティパッチ自動適用
    • systemd hardening: ProtectSystem=strict, NoNewPrivileges=yes 等

    10. 独立アラート¶

    バックアップノードはPVEの監視スタック（Prometheus/Grafana）に依存しない独自の通知経路を持つ:

    • Discord Webhook / LINE Notify（outbound HTTPS）
    • 成功通知（dead man's switch: 3日連続で来なければ異常）
    • 失敗・異常検知時の即時通知

    11. 管理操作の物理分離（端末感染対策）¶

    ゲーミングPC（VLAN 10）にTailscale・SSH鍵・管理用ブラウザセッションが集中しており、感染時に全VLAN分離がバイパスされる構造的欠陥がある。

    対策: 管理端末の物理分離

    • ゲーミングPCからTailscaleを削除、SSH鍵・管理系認証情報を除去
    • 管理専用の物理端末（古いPC/ノートPC）をVLAN 90に接続
    • 管理端末にのみTailscale、SSH鍵、YubiKey、ブラウザ管理セッションを配置
    • Tailscale ACLで管理端末以外からの管理アクセスを拒否

    これによりゲーミングPC（高リスク環境: Steam Mod, Discord等）が感染しても、到達可能な宛先はゲームサーバーポート+Internetのみに限定される。

    12. 会社PCの接続ポリシー¶

    会社PCを自宅WiFiに接続する場合はVLAN 31 (Guest) を使用する。VLAN 20 (Home) に接続すると、会社VPNとの橋渡しにより会社セキュリティインシデントが自宅ラボに波及するリスクがある。

    ハードウェア選定¶

    選択肢 コスト 利点 欠点
    Raspberry Pi 5 (4GB) ~¥18,000 ARM多様性、5W省電力 Docker ARM互換性（不使用なので影響なし）
    中古ミニPC (x86) ~¥10,000-15,000 Docker完全互換（不使用）、内蔵SSD カーネル多様性なし
    手持ちの古いPC/ノートPC ¥0 無料、ノートPCならバッテリーがUPS代わり 消費電力高

    推奨: Raspberry Pi 5 + USB SSD。ARMアーキテクチャによるx86との多様性確保がセキュリティ上最大の利点。

    結果¶

    • 同時障害耐性: GitHub全漏洩 + PVE全台カーネルパニックの同時発生でも、バックアップノードから3時間以内で完全復旧可能
    • ランサムウェア耐性: PULL型 + inbound ゼロ + append-only により、Lab内からバックアップノードを破壊する経路が存在しない
    • スロー・ポイズニング耐性: 整合性検証 + 長期保持 (weekly/monthly) + R2 Object Lockの多層防御
    • VLAN 90 修正: 管理VLANの全許可ルールを廃止し、ゼロトラスト原則を管理セグメントにも適用
    • コスト: ~¥18,000（Raspberry Pi 5 + USB SSD）+ R2/GitHub無料枠
    • 運用負荷: 全自動（systemd timer）、異常時のみ通知
    • 端末感染耐性: 管理操作を物理分離し、ゲーミングPC感染時のVLANバイパスを防止
    • 会社PC分離: Guest VLAN接続ポリシーにより、会社インシデントの波及を防止
