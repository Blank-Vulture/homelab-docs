    ADR-010: AI自律管理基盤 — Vault・Teleport・Authentik統合によるゼロ長命鍵アーキテクチャ¶

    ステータス¶

    提案中（段階移行: Phase 1〜3）

    設計原則（このADRの核心）¶

    「SOPS+age+YubiKeyは、HA認可基盤の完全なフォールバックである。管理端末への依存は非常用復旧に限定する。」

    通常運用において管理端末は不要。Vault・Teleport・Authentik の HA クラスタが認可基盤として常時稼働する。 これらが全滅したときの break-glass パスとして SOPS+age+YubiKey が機能する。

    通常運用:  AI/人間 → Vault/Teleport/Authentik (HA, 自律)
               ↓ 全コンポーネント障害時のみ
緊急復旧:  人間 → 管理端末 + YubiKey → SOPS → 直接SSH/kubectl
               ↓ 管理端末も到達不能な場合
最終手段:  人間 → OOB端末 (VLAN 91/101, ADR-007)

    コンテキスト¶

    現状の制約¶

    現在のシークレット・アクセス管理は「人間がYubiKeyをタッチして操作する」前提で設計されている。

    現状の仕組み 制限
    SOPS + age + YubiKey CI/AI が操作するたびに YubiKey タッチが必要。自律実行不可。
    長命 SSH 鍵（RUNNER_SSH_KEY） 漏洩時の影響範囲が広い。鍵ローテーションが手動。
    静的 kubeconfig（SOPS 暗号化） 有効期限なし。配布先の管理が煩雑。
    平文パスワード（SOPS 暗号化） DB・サービス間は静的認証情報。自動ローテーション不可。
    CI 秘密鍵（Forgejo Secret） 長命鍵。Forgejo 侵害時に全ホストに到達される。

    目標とする状態¶

    AIエージェント（Claude Code / Forgejo Actions）がYubiKeyなしで安全に以下を実行できる：

    • 新規 LXC / VM のプロビジョニング（Proxmox API 認証）
    • SSH でのホスト設定（Ansible / 直接操作）
    • kubectl でのワークロード管理
    • DB 接続・マイグレーション
    • シークレットの読み書き・ローテーション

    かつ、すべての操作が 監査ログに記録され、最小権限で実行され、証明書は短命（自動失効）であること。

    決定¶

    以下の3コンポーネントを段階的に導入し、現状の SOPS+age+YubiKey 体制を置き換える。

    コンポーネント概要¶

    ┌─────────────────────────────────────────────────────────────────┐

│  すべてのサービスの認証ハブ。OIDC/SAML プロバイダー。           │

└────────────────────┬────────────────────────────────────────────┘

│ SSH CA証明書   │     │ 動的シークレット          │
│ kubectl プロキシ│    │ PKI/SSH CA               │

         ・短命証明書のみ保持
         ・操作ログ完全記録
         ・ロールに基づく最小権限

    アクセスモデル（AIエージェント）¶

    AI が SSH したいとき:
  1. Vault AppRole で認証 → 一時トークン取得 (TTL: 1時間)
  2. Vault SSH CA に署名要求 → 証明書取得 (TTL: 10分)
  3. Teleport Machine ID 経由で対象ホストに接続
  4. セッションが Teleport に録画・監査ログ記録
  5. 証明書は TTL 後に自動失効

AI が DB にアクセスしたいとき:
  1. Vault AppRole 認証 → 一時トークン
  2. Vault Database Secret Engine → 動的クレデンシャル発行 (TTL: 15分)
  3. 接続完了後、クレデンシャルは自動失効

    移行フェーズ¶

    Phase 1: HashiCorp Vault 導入（SOPS 移行）¶

    目的: 静的シークレットの動的管理への移行。CI/AI が YubiKey なしでシークレットにアクセスできる状態を実現。

    実装内容:

    • Vault を Lab クラスタ（k3s）に HA 構成でデプロイ（Raft バックエンド）
    • Vault Unseal: YubiKey FIDO2 でシールド解除（人間操作のみ、起動時1回）
    • auto-unseal のバックアップとして Shamir's Secret Sharing（閾値: 3/5）
    • 認証方式:
    • AI/CI: AppRole（Role ID + Secret ID。Secret ID は使い捨て）
    • 人間: Authentik OIDC（Phase 3 で統合）
    • シークレット移行:
    • secrets/*.sops.yml → Vault KV v2 に段階移行
    • SOPS は非常用バックアップとして維持（emergency recovery）
    • Proxmox 認証: Vault が Proxmox API トークンを管理・ローテーション
    • Forgejo Actions: RUNNER_SSH_KEY 廃止 → Vault Agent Injector で短命証明書を払い出し

    完了条件: - Forgejo Actions が RUNNER_SSH_KEY Secret を使わずにホストへの SSH が可能 - sops -d を使うスクリプトが Vault CLI に置き換わっている - 全シークレットに TTL と自動ローテーションが設定されている

    Phase 2: Teleport 導入（SSH/kubectl 統合）¶

    目的: すべての SSH・kubectl アクセスを Teleport 経由に統一し、監査ログを確保。

    実装内容:

    • Teleport を DMZ クラスタ（PVE-DMZ-1）にデプロイ
    • DMZ に置くことで内部クラスタ障害時も Teleport 管理が継続
    • SSH 証明書 CA:
    • Vault SSH Secret Engine を Teleport の CA として構成
    • 各 PVE / LXC / VM の authorized_keys を廃止 → Teleport CA 公開鍵のみ信頼
    • 証明書 TTL: 人間 = 8時間、AI/CI = 10分
    • kubectl アクセス:
    • k3s kubeconfig 廃止 → tsh kube login k3s-cluster に移行
    • AI は k8s ServiceAccount + Teleport RBAC で最小権限
    • DB アクセス:
    • Teleport DB プロキシ + Vault Dynamic Credentials の組み合わせ
    • PostgreSQL への直接アクセス不可 → Teleport プロキシ必須
    • PVE OOB: OOB 管理端末（VLAN 91/101）は Teleport 管理外を維持（ADR-007 エアギャップ原則）

    完了条件: - すべての SSH セッションが Teleport に録画されている - ~/.ssh/authorized_keys が全ホストから削除されている - kubectl が Teleport 経由でのみ動作している

    Phase 3: Authentik 導入（IdP 統合）¶

    目的: 全サービスの認証を Authentik に集約し、シングルサインオンと統一ポリシーを実現。

    実装内容:

    • Authentik を Lab クラスタ（k3s）にデプロイ
    • OIDC プロバイダーとして統合するサービス:
    • Forgejo: Authentik OIDC でログイン
    • Vault: Authentik OIDC 認証（人間ログイン）
    • Teleport: Authentik OIDC 認証
    • Grafana: Authentik OIDC
    • Immich: Authentik OIDC（家族アカウント管理）
    • AI Service Account:
    • Authentik で AI エージェント用サービスアカウントを作成
    • JWT トークン → Vault/Teleport での認証に使用
    • スコープ: 操作ロール（provisioner / operator / reader）ごとに分離
    • MFA ポリシー:
    • 人間: YubiKey FIDO2 必須（すべての管理操作）
    • AI: Service Account（MFA 不要、代わりに TTL と監査で保証）
    • Passkey 対応: 将来的な YubiKey → Passkey 移行の布石

    完了条件: - 全サービスへのログインが Authentik 経由に統一されている - AI の操作ロールが Authentik 管理下にある

    AI 完全自律オペレーションの例（Phase 3 完了後）¶

    例: AI が新規 LXC をプロビジョニングするとき

1. Vault AppRole 認証 → トークン取得（TTL: 1h）
2. Vault から Proxmox API トークンを動的発行
3. OpenTofu で LXC 定義を生成・apply（Vault トークン使用）
4. Vault SSH CA から証明書取得（TTL: 10min）
5. Teleport Machine ID で LXC に SSH
6. Ansible で設定適用（証明書認証）
7. Vault DB Secret Engine からサービス用クレデンシャル発行
8. k3s に Deployment apply（Teleport kubectl プロキシ経由）
9. 全ステップの操作ログが Teleport + Vault Audit に記録
10. すべての認証情報は TTL 後に自動失効

AI が保持し続ける長命な鍵: なし

    代替案と不採用理由¶

    代替案 不採用理由
    SOPS + software age key を CI に配布 長命鍵。漏洩時の影響範囲が変わらない
    Bitwarden Secrets Manager OSS セルフホストが不安定。Vault の機能（SSH CA・動的シークレット）がない
    1Password Secrets Automation 有料クラウド前提。エアギャップ環境での動作不保証
    AWS Secrets Manager / GCP Secret Manager クラウド依存。ホームラボの自律性原則に反する
    Infisical Vault より機能が限定的。SSH CA が未実装
    Teleport を使わず Vault SSH CA のみ セッション録画・kubectl プロキシ・DB プロキシが欠ける

    セキュリティ上の考慮事項¶

    Vault の高可用性・耐障害性¶

    • Vault が落ちると AI の全自律操作が停止 する。単一障害点になり得る。
    • 対策: Vault Raft HA（3ノード）+ 定期スナップショット + SOPS バックアップの維持

    Vault 侵害シナリオ¶

    • Vault の AppRole Secret ID が漏洩 → 攻撃者が AI と同等の権限を取得
    • 対策: Secret ID の使い捨て（Use-Limit: 1）+ Response Wrapping + Teleport 監査ログで異常検知

    Teleport 侵害シナリオ¶

    • Teleport Proxy の侵害 → セッション録画の改ざん可能性
    • 対策: Teleport を DMZ（ADR-005）に配置。コアネットワークから物理分離。

    OOB 管理との関係（ADR-007）¶

    • Vault/Teleport が全滅した場合、AI は操作不能になる
    • OOB 管理端末（VLAN 91/101）は Vault/Teleport 非依存で動作（ADR-007 原則維持）
    • SOPS + age + YubiKey は OOB 復旧時の emergency 手段として保持

    管理端末依存の最小化（このADRの核心的目標）¶

    Phase 3 完了後、管理端末（VLAN 90）が必要な操作は以下のみとする：

    操作 管理端末が必要な理由 代替
    Vault 初回 Unseal YubiKey タッチ必須 Vault HA で再起動頻度を最小化
    Vault unseal key 再構成 Shamir 閾値 3/5 鍵保持者（家族含む）が分散保持
    OOB 端末の物理操作 エアギャップ原則（ADR-007） 変更なし
    SOPS break-glass 復旧 認可基盤全滅時のみ 年1回以下を目標

    通常の運用・メンテナンス・AI 自律操作で管理端末を使う場面はゼロ を達成目標とする。

    HA 設計要件¶

    Vault/Teleport/Authentik が「完全な認可基盤」であるためには、以下の HA が必要：

    コンポーネント HA 構成 目標 RTO
    Vault Raft 3ノード（k3s StatefulSet） < 30秒（自動フェイルオーバー）
    Teleport Auth Server 2ノード + Proxy 2ノード < 60秒
    Authentik Worker 2ポッド + PostgreSQL HA < 30秒
    PostgreSQL（共有バックエンド） Patroni / CloudNativePG HA < 30秒

    これらが達成されてはじめて、SOPS+age が「フォールバック」と呼べる状態になる。
